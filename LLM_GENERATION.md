# LLM Authorship Disclosure

This document discloses the LLM authorship of `muskitty-css-parser`,
in accordance with This Week in Rust's submission policy.

## Summary

The CSS parser logic in this crate was generated by an LLM-based
coding agent (GLM-5.2 via TRAE IDE), supervised by a human developer.
The crate was extracted from `muskitty-css` (commit bbcbaab in the
workspace root) as a standalone crate to allow reuse by syntax
highlighters, CSS linters, and selector parsers without dragging in
the full CSS module.

## Origin

This crate was originally developed inside `muskitty-css` as the
`src/parser/` module. See
[muskitty-css/LLM_GENERATION.md](https://github.com/Ink-dark/MusKitty/blob/main/crates/muskitty-css/LLM_GENERATION.md)
for the full LLM authorship disclosure covering the parser's
implementation history, test-feedback loop, and spec citation discipline.

## What the LLM Did

1. **Implemented the parser** (CSS Syntax §5) — all 8 entry points
   (§5.4) and 13 consume algorithms (§5.5) of the CSS parser,
   including:
   - §5.4 entry points (stylesheet / rule list / rule / declaration
     / declaration list / component value / list of component values /
     comma-separated list)
   - §5.5.1 consume a stylesheet's contents
   - §5.5.2 consume a list of rules (with `CDO`/`CDC` skipping for
     top-level rule-lists)
   - §5.5.3 consume an at-rule (prelude + block)
   - §5.5.4 consume a qualified rule (prelude + block)
   - §5.5.5 consume a declaration (name + value, with `!important`
     flag extraction)
   - §5.5.6 consume the remnants of a bad declaration (recovery)
   - §5.5.7 consume a list of component values (block-aware)
   - §5.5.8 consume a simple block (with `(` / `[` / `{` matching
     and end-token enforcement)
   - §5.5.9 consume a block's contents (declarations + at-rules +
     qualified-rules-as-declarations fallback)
   - §5.5.10 consume a component value (token / block / function
     dispatch)
   - §5.5.11 consume a function (name + argument list)
   - §5.5.12 consume a unicode-range value (§4.3.14 integration with
     the §5.4.6 rule entry point)

2. **Diagnosed and fixed its own bugs** — when the test harness
   reports a failure, the agent reads the failing input, the expected
   output, and the relevant spec section, then proposes a fix.

3. **Selected Rust idioms** — the agent chose `TokenStream` as a
   thin wrapper over `CssTokenizer` with `peek`/`next`/`reconsume`
   primitives, `Option<ParseError>` for soft-error reporting, and
   exhaustive `match` on `ComponentValue` variants. These choices
   were not directed by the human.

## What the Human Did

1. **Authored the project constitution** — three hard rules that the
   agent must follow at all times:
   - CSSWG spec is ground truth (spec > WPT > Chromium when they conflict)
   - Safe Rust only (zero `unsafe`, zero C/C++ deps)
   - Spec-compliant, not test-compliant (never modify a test to pass it
     unless the spec proves the test is wrong, with chapter citation)

2. **Built the test-feedback loop** — a harness that runs
   `cargo test`, captures per-test pass/fail, and feeds failing
   cases back to the agent with the expected AST and the
   relevant CSS Syntax spec section.

3. **Reviewed every commit** — the human reads every diff before
   `git commit`.

## Test Results

- **Parser**: 62/62 tests pass (55 integration tests across 6 files +
  7 doctests), covering:
  - Entry points: stylesheet/rule/declaration/component-value/list parsing
  - Consume algorithms: at-rule, qualified rule, declaration,
    simple block, block's contents, function, unicode-range
  - Bad-declaration recovery (`;` and `}` termination)
  - Bad-block recovery (unbalanced `{`/`}`/`(`/`[`)
  - `!important` flag detection
  - Token preservation in `ComponentValue::PreservedToken`
  - Whitespace handling at the boundaries of component value lists
  - Comma-separated list with leading-whitespace preservation
    per §5.4.10 (a frequently missed spec detail)
